# C++ File-Based Chat Application - Project Documentation

## 📄 1. Project Overview

**Project Title:** File-Based Chat Application (Version 2.0)
**Objective:** To develop a robust, console-based (CLI) chat application in C++ that facilitates secure communication between users through a centralized file-storage system.
**Problem Statement:** Conventional command-line chat applications often lack persistent storage, proper memory management, and structured object-oriented designs, leading to fragile systems that cannot retain state across sessions or scale efficiently.
**Solution Approach:** This project leverages advanced Object-Oriented Programming (OOP) paradigms and the C++ Standard Template Library (STL) to build a robust system. It incorporates secure user authentication, role-based access control (Admin vs. Member), encrypted private messaging, and persistent state management using file I/O operations.

---

## 🧠 2. System Design

### Overall Architecture
The application is built on a monolithic architecture tailored for desktop execution, completely localized to the user's machine. The system state is actively maintained in memory using STL containers and synchronized with the disk to ensure persistence.

### Module Breakdown
1. **Core Module (Entities):** Defines the foundational entities such as `User` and `Message`, abstracting their generalized behaviors.
2. **Authentication Module:** Handles user registration, role allocation, and login verification.
3. **Chat Module:** Orchestrates the `PrivateChat` logic, mapping one-to-one user conversations.
4. **Data Persistence Module:** The `FileManager` handles reading and writing application state (users, chats) to the disk.
5. **Utility Module:** Includes the `SearchEngine` for querying chat history, `EncryptionManager` for securing messages, and a template-based `Logger` for auditing.

### How Components Interact
The `main.cpp` acts as the system controller. When a user sends a message, a `PrivateChat` instance delegates the creation of an `EncryptedMessage`. The `PrivateChat` then stores the message in memory. Concurrently, the `FileManager` serializes the chat state to a text file. The `Logger` runs asynchronously to log major events (e.g., login, registration).

## 🧱 3. Class Design

### `User` (Abstract Base Class)
*   **Purpose:** Serves as the blueprint for all system users.
*   **Attributes:** `username`, `password`, `isOnline`, `blockedUsers` (STL `set`), `loginAttempts`.
*   **Methods:** `login()`, `logout()`, `blockUser()`, and pure virtual methods `displayProfile()` and `canModerate()`.
*   **Design Decisions:** Abstracting the `User` class ensures that raw users cannot be instantiated. Roles (Admin/Member) must be explicitly defined.

### `Admin` & `Member` (Derived Classes)
*   **Purpose:** Implements specific role behaviors.
*   **Attributes:** Inherits from `User`.
*   **Methods:** `Admin` includes `removeUserFromRoom()` and `muteUser()`. `Member` includes `changePassword()`.
*   **Design Decisions:** Separating these classes enforces strict Role-Based Access Control (RBAC) utilizing runtime polymorphism.

### `Message` (Abstract Base Class), `TextMessage`, & `EncryptedMessage`
*   **Purpose:** Represents a single communication unit.
*   **Attributes:** `sender`, `content`, `timestamp`, `isRead`.
*   **Methods:** `display()`, `getType()` (pure virtual). `EncryptedMessage` adds `encrypt()` and `decrypt()`.
*   **Design Decisions:** `EncryptedMessage` utilizes a Caesar Cipher approach, ensuring that message content written to the file system is not stored in plaintext.

### `PrivateChat`
*   **Purpose:** Manages the context of a 1-on-1 conversation.
*   **Attributes:** `id`, `user1`, `user2`, `messages` (STL `vector` of `Message*`).
*   **Methods:** `sendMessage()`, `viewHistory()`, `markAsRead()`.
*   **Design Decisions:** Chat IDs are generated by sorting and concatenating the two usernames (e.g., "Alice_Bob"), ensuring a unique and consistent file identifier regardless of who initiates the chat.

### `FileManager`
*   **Purpose:** Abstracts all File I/O operations.
*   **Attributes:** Paths to `users.txt` and the `private_chats/` directory.
*   **Methods:** `saveUsers()`, `loadUsers()`, `savePrivateChat()`, `loadPrivateChat()`.
*   **Design Decisions:** Centralizing file logic prevents file-locking collisions and isolates disk operations from core business logic.

### `SearchEngine`
*   **Purpose:** Allows querying of message histories.
*   **Methods:** Overloaded `search()` methods.
*   **Design Decisions:** Separating search into its own engine adheres to the Single Responsibility Principle (SRP).

### `Logger` (Template Class)
*   **Purpose:** Handles system auditing.
*   **Methods:** `log(T message)`.
*   **Design Decisions:** Implemented as a C++ Template to allow logging of various data types (strings, custom objects via overloaded operators) without code duplication.

## ⚙️ 4. Feature Implementation

### Authentication
*   **Register:** Prompts for credentials and role. Dynamically allocates an `Admin` or `Member` object and pushes it to the `users` vector. Calls `FileManager::saveUsers()` to persist data.
*   **Login:** Iterates over the `users` vector comparing credentials. Sets a `currentUser` pointer to track session state if successful.

### Private Chat
*   **Message Sending:** Users instantiate an `EncryptedMessage`. The system captures the current timestamp, pushes the message pointer to the `PrivateChat::messages` vector, and flushes the entire chat to a specific file.
*   **Chat Retrieval:** Chat files are loaded lazily. If a chat file exists for "UserA_UserB", it is deserialized into a `PrivateChat` object and placed in the active memory map.

### File Handling
*   **User Storage:** Stored in `data/users.txt`.
*   **Chat Storage:** Stored in `data/private_chats/<chat_id>.txt`.
*   **File Structure Design:** Plain text formats delimited by special characters or newlines to distinguish object properties (e.g., sender, timestamp, ciphertext) allowing for rapid parsing.

### Search Feature
*   **How Search Works:** Iterates through the loaded vectors of `Message*` objects.
*   **Algorithms Used:** Uses C++ STL `<algorithm>` methods and `std::string::find()` to perform substring matching against the message content.
*   **STL Usage:** Highly reliant on `std::vector` iterators to efficiently parse large sets of messages in memory.

## 🧩 5. OOP Concepts Implementation

*   **Encapsulation:** Class attributes (e.g., `password`, `messages`) are marked `private` or `protected`. State is modified strictly through setter and getter methods, preventing unauthorized mutations.
*   **Inheritance:** `Admin` and `Member` inherit the baseline structure of `User`, while `TextMessage` and `EncryptedMessage` inherit from `Message`, promoting code reuse.
*   **Polymorphism (Runtime):** The `main.cpp` handles a vector of `User*`. Calling `u->displayProfile()` correctly resolves to the `Admin` or `Member` implementation using vtables (virtual functions).
*   **Polymorphism (Compile-Time):** The `SearchEngine` class uses method overloading. Calling `search()` with two arguments searches by keyword, while providing three arguments filters by keyword *and* sender.
*   **Abstraction:** The `User` and `Message` classes contain pure virtual functions (e.g., `virtual ~User() = 0;`), making them abstract interfaces that cannot be instantiated directly.
*   **Composition:** `PrivateChat` is composed of multiple `Message` objects. `FileManager` acts dynamically upon these composed structures.

---

## 📚 6. STL Usage

*   **`std::vector`:** Used extensively for managing collections of users (`vector<User*>`) and messages (`vector<Message*>`). Chosen for its contiguous memory allocation and O(1) random access performance.
*   **`std::map`:** Used to cache loaded private chats (`map<string, PrivateChat*>`). The string key allows O(log n) lookups for active chat sessions based on their generated Chat ID.
*   **`std::set`:** Used in `User` to track `blockedUsers`. Sets inherently prevent duplicates and offer O(log n) lookup times for checking block status.
*   **`<algorithm>`:** Utilized functions like `std::find_if` and `std::swap` to sort usernames for chat IDs and rapidly search user lists, reducing boilerplate loop code.

---

## 🔐 7. Advanced Features

*   **Operator Overloading:** The `<<` operator is heavily overloaded. For instance, `friend ostream& operator<<(ostream& os, const User& u)` allows developers to simply call `cout << *userObject;` to print a formatted profile.
*   **Templates:** The `Logger<T>` class template allows the system to flexibly write standard strings, integers, or complex objects directly to log files.
*   **Type Conversion:** The `User` class implements a custom type conversion operator (`operator std::string() const`). This allows a `User` object to be implicitly evaluated as a string (yielding their username) when passed to the logger.
*   **Exception Handling:** Input validation wraps sensitive operations in `try-catch` blocks. For example, attempting to send an empty message throws an `std::invalid_argument` exception, which is gracefully caught and displayed to the user without crashing the CLI.

---

## 💾 8. File System Design

*   **File Formats:** Data is structured in standard text files (`.txt`).
*   **Data Storage Logic:**
    *   `data/logs/`: Contains chronological append-only `.log` files.
    *   `data/users.txt`: Contains user credentials and roles.
    *   `data/private_chats/`: Contains individual chat ledgers.
*   **Read/Write Workflow:** On startup, `FileManager` reads `users.txt` to populate memory. Chat files are read dynamically on-demand when a user initiates a chat. Writes are performed synchronously following any state-mutating action (registration, message send, user deletion).

---

## 🖥️ 9. User Interface (CLI)

*   **Menu System:** A hierarchical textual interface. Root menu routes to Authentication; authenticated state routes to a Dashboard; Dashboard routes to targeted sub-menus (Admin Menu, Private Chat Menu).
*   **Navigation Flow:** Driven by infinite `while` loops containing conditional branching based on integer inputs.
*   **Input Handling:** Standard `std::cin` is used. A dedicated `clearInput()` function flushes the input buffer utilizing `numeric_limits<streamsize>::max()` to prevent infinite loops if a user inputs a `char` when an `int` is expected.

---

## ⚠️ 10. Error Handling

*   **File Errors:** If `FileManager` fails to open a target file or directory, the application falls back to memory-only structures or uses shell commands (`mkdir`) at startup to ensure the environment is prepped.
*   **Invalid Input:** Type-mismatches in `cin` are caught, the stream is cleared, and users are re-prompted cleanly.
*   **Edge Cases:** Deleting a user iteratively scrubs the active `map` and file system to ensure no dangling pointers or orphaned chat files remain, preventing undefined behavior.

---

## 🚀 11. Challenges and Solutions

*   **Challenge:** Managing pointer lifecycles in C++ and preventing memory leaks when deleting users or closing the app.
    *   **Solution:** Implemented rigorous cleanup loops iterating through vectors and maps, calling `delete` on dynamically allocated pointers in both `main.cpp` and specific destructors (e.g., `~PrivateChat`).
*   **Challenge:** Dangling pointers when an Admin deletes a user who has active chat histories.
    *   **Solution:** The user deletion routine includes a secondary pass over the `privateChats` map, safely erasing and deleting any `PrivateChat` objects linked to the removed user's username.

## 📈 12. Future Improvements

*   **Scalability Ideas:**
    *   Transition from raw pointers to Smart Pointers (`std::unique_ptr` and `std::shared_ptr`) to eliminate manual memory management overhead.
    *   Migrate disk I/O from raw text files to a lightweight embedded database (like SQLite) to improve query speed.
*   **Possible Upgrades:**
    *   Implementing Multithreading using `std::thread` to allow asynchronous message receiving without blocking the `cin` prompts.
    *   Adding group chat functionalities using the Observer pattern.
